## D1.5단계: 인덱스 설정 & 스토리지 전략

> **학습 목표**: 인덱스 레벨 설정(refresh, translog, merge policy)이
> 성능/비용/안정성에 미치는 영향을 이해하고, 데이터 특성에 맞는 최적 설정 조합을 결정할 수 있다.
> 또한 Hot-Warm-Cold 계층 설계와 멀티테넌시 routing 전략을 실전에 적용할 수 있다.
>
> **핵심 질문**: "이 인덱스의 설정을 어떻게 잡아야 비용 대비 최적 성능이 나오는가?"
>
> **전제 조건**: D1(샤드 사이징) 완료

---

### D1.5-1. 인덱스 설정 튜닝 프레임워크

```
인덱스 설정 = 성능/비용/안정성의 트레이드오프 조합

핵심 설정 카테고리:
  1. Refresh (검색 가시성 지연 vs 색인 처리량)
  2. Translog (내구성 vs 색인 속도)
  3. Merge Policy (검색 성능 vs CPU/IO 부하)
  4. Store (압축 vs CPU)
  5. 기타 (codec, max_result_window 등)
```

---

### D1.5-2. Refresh 설정 심화

```
refresh_interval: 메모리 버퍼 → 검색 가능 Segment 변환 주기

| 값 | 효과 | 적합한 경우 |
|----|------|-----------|
| "1s" (기본) | NRT, 색인 후 1초 내 검색 가능 | 실시간 검색 필요 (대시보드) |
| "5s"~"30s" | 색인 처리량 향상, Segment 생성 감소 | 로그 수집 (수초 지연 허용) |
| "-1" (비활성) | 수동 refresh 전까지 검색 불가 | 대량 초기 로딩, reindex |

트레이드오프 상세:
  refresh_interval 길게 → 
    ✓ 색인 TPS 향상 (버퍼 flush 빈도 감소)
    ✓ Segment 수 감소 (merge 부하 감소)
    ✗ 검색 지연 증가 (새 문서가 안 보임)
    ✗ 메모리 버퍼 점유 증가

프로젝트별 권장:
  - 로그 인덱스: "5s"~"30s" (초 단위 정확성 불필요)
  - Issue History: "1s" (UI에서 즉시 반영 기대)
  - 대량 마이그레이션 중: "-1" → 완료 후 POST /_refresh
```

### D1.5-3. Translog 설정 심화

```
Translog = WAL(Write-Ahead Log). 데이터 내구성 보장.

핵심 설정:

index.translog.durability:
  - "request" (기본): 매 색인 요청마다 fsync → 데이터 유실 0
  - "async": 주기적으로 fsync → 색인 빠르지만 장애 시 최대 sync_interval 만큼 유실 가능

index.translog.sync_interval: "5s" (async일 때 fsync 주기)
index.translog.flush_threshold_size: "512mb" (이 크기 넘으면 flush)

| durability | 색인 속도 | 데이터 안전 | 적합한 경우 |
|-----------|---------|---------|-----------|
| request | 보통 | 완벽 | 중요 데이터 (Issue History) |
| async | 빠름 | 최대 5초 유실 가능 | 로그 (유실 허용 가능) |

저렴한 티어 전략:
  - 로그 → async (5s 유실 허용, 색인 부하 감소)
  - Issue History → request (유실 불가)
```

### D1.5-4. Merge Policy 설정

```
Segment Merge = 여러 작은 Segment를 합치는 백그라운드 작업

왜 중요한가:
  - Segment 많으면 → 검색 느림 (모든 Segment 탐색)
  - Merge 중 → CPU/IO 소비 (색인/검색과 경쟁)
  - 저렴한 티어에서 과도한 Merge = 클러스터 불안정

핵심 설정:

index.merge.policy.max_merge_at_once: 10 (기본)
  → 한 번에 합칠 최대 Segment 수

index.merge.policy.segments_per_tier: 10 (기본)
  → 티어당 유지할 Segment 수 (낮추면 Merge 빈도 증가)

index.merge.policy.max_merged_segment: "5gb" (기본)
  → 이 크기 이상의 Segment는 Merge 대상에서 제외

index.merge.scheduler.max_thread_count: 1 (권장, 저렴한 티어)
  → Merge 스레드 수 (SSD 아닌 EBS에서는 1이 안전)

튜닝 전략:
  - 저렴한 티어: max_thread_count=1, 공격적 Merge 피하기
  - 쓰기 완료된 인덱스: force_merge로 Segment 1개로 합침
  - 활성 쓰기 중: 기본값 유지 (ES 자동 판단 신뢰)
```

### D1.5-5. 압축 설정 (Codec)

```
index.codec:
  - "default" (LZ4): 빠른 압축/해제, 적당한 압축률
  - "best_compression" (DEFLATE): 높은 압축률, 느린 해제

| 코덱 | 압축률 | 색인 속도 | 검색 속도 | 적합한 경우 |
|------|--------|---------|---------|-----------|
| default (LZ4) | 보통 | 빠름 | 빠름 | 활성 인덱스, 검색 빈번 |
| best_compression | 높음 (20~30% 절약) | 느림 | 느림 | Cold 인덱스, 디스크 절약 |

저렴한 티어 전략:
  - Hot 인덱스: default (검색 성능 우선)
  - Warm/Cold 인덱스: best_compression (디스크 절약 우선)
  - ISM으로 Warm 전환 시 codec 변경 (force_merge와 함께)
```

### D1.5-6. Hot-Warm-Cold 실전 설계

```
단일 티어 (저렴한 환경):
  → Hot-Delete만 사용 (Warm/Cold 노드 없음)
  → ISM: 기간 경과 → 삭제
  → best_compression은 삭제 전 마지막 단계에서만 의미 있을 수 있음

다중 티어 (확장된 환경):
  Hot:
    - 인스턴스: r6g.large (메모리 중심)
    - 스토리지: gp3 SSD
    - 데이터: 최근 1~3일
    - 설정: refresh=1s, codec=default, replica=1

  Warm:
    - 인스턴스: r6g.medium 또는 UltraWarm
    - 스토리지: gp3 또는 S3 (UltraWarm)
    - 데이터: 3~30일
    - 설정: refresh=-1, codec=best_compression, replica=0, read_only

  Cold:
    - 인스턴스: 최소 또는 UltraWarm
    - 데이터: 30~90일
    - 설정: 검색 드묾, 보관 목적

비용 시뮬레이션 프레임워크:
  월간 비용 = Σ (인스턴스 비용 + EBS 비용) × 각 티어
  
  예시 (ap-northeast-2):
  | 티어 | 인스턴스 | EBS | 보존기간 | 데이터량 | 월 비용 |
  |------|---------|-----|---------|---------|--------|
  | Hot | r6g.large×2 | gp3 100GB×2 | 3일 | ~12GB | ~$XXX |
  | Warm | UltraWarm medium×2 | S3 | 27일 | ~108GB | ~$YYY |
  | 합계 | | | 30일 | 120GB | ~$ZZZ |

  vs 단일 티어:
  | Hot only | r6g.large×2 | gp3 500GB×2 | 30일 | 120GB | ~$WWW |

  → 어느 시점부터 Hot-Warm 분리가 비용 이점이 있는지 계산
```

### D1.5-7. 인덱스 분할 전략 심화

**시간 기반 분할 (Time-based)**:
```
패턴: logs-YYYY-MM-DD 또는 Data Stream

선택 기준:
  - 일간 데이터 > 5GB → 일별 분할
  - 일간 데이터 1~5GB → 일별 분할 (shard 1개)
  - 일간 데이터 < 1GB → 주별 또는 월별 분할
  - 일간 데이터 < 100MB → 월별 분할 또는 단일 인덱스

보존 기간과의 관계:
  총 인덱스 수 = 보존기간 / 분할주기
  총 샤드 수 = 총 인덱스 수 × 인덱스당 shard × (1 + replica)
  
  제약: 노드당 600~1000 shard 이내 (공식 권장)
```

**크기 기반 분할 (Size-based, Rollover)**:
```
패턴: Data Stream + rollover 조건

장점: 인덱스 크기 균일 → 샤드 크기 예측 가능
단점: 시간 범위가 불균일 (피크 시 빠르게 rollover)

적합: 트래픽 변동이 큰 서비스 (피크 10배 이상)
```

**엔티티 기반 분할 (Entity-based)**:
```
패턴: issues-team-a, issues-team-b 또는 routing

사용 케이스:
  - 멀티테넌시 (고객/팀별 격리)
  - 데이터 접근 패턴이 엔티티 단위로 격리됨
  - 삭제 단위가 엔티티 (고객 탈퇴 시 인덱스 통째 삭제)
```

### D1.5-8. Routing 전략 (멀티테넌시)

```
기본 routing: shard = hash(_id) % number_of_shards
  → 문서가 모든 샤드에 균등 분포

커스텀 routing: shard = hash(routing_value) % number_of_shards
  → 같은 routing 값을 가진 문서가 같은 샤드에 집중

PUT /issues/_doc/ISS-001?routing=team-backend
{ "team": "backend", "title": "..." }

GET /issues/_search?routing=team-backend
{ "query": { "term": { "team": "backend" } } }
→ 1개 샤드만 검색 (전체 샤드 fan-out 불필요 → 빠름)

장점:
  - 특정 엔티티 검색 시 1개 샤드만 조회 (성능 향상)
  - 관련 데이터가 물리적으로 같은 위치 (join 대체 가능)

단점/위험:
  - 샤드 불균형 (특정 routing에 데이터 편중)
  - routing 값 변경 불가 (문서 삭제 후 재색인 필요)
  - routing 지정 잊으면 잘못된 샤드에 저장

설계 원칙:
  - routing 값의 cardinality가 shard 수보다 충분히 커야 균등 분포
  - routing 값별 데이터량이 비슷해야 함
  - 저렴한 티어(shard 1개)에서는 routing 의미 없음
```

### D1.5-9. 데이터 보존 vs 비용 의사결정 매트릭스

```
의사결정 입력:
  1. 데이터 유형 (로그 / 메트릭 / 이벤트 / 엔티티)
  2. 법적/규정 보존 요구 (GDPR, 감사 로그 등)
  3. 비즈니스 조회 빈도 (최근 N일만 활발 vs 과거도 자주)
  4. 디스크 비용 제약

의사결정 매트릭스:

| 데이터 유형 | 활발 조회 기간 | 규정 보존 | 전략 |
|-----------|-------------|---------|------|
| 앱 로그 | 7일 | 없음 | Hot 7일 → Delete |
| 감사 로그 | 30일 | 1년 | Hot 30일 → Warm 11개월 → Delete |
| 메트릭 | 3일 | 없음 | Hot 3일 → Delete (또는 다운샘플링) |
| Issue History | 무기한 | 없음 | Hot 유지 (크기 작음) |
| 알림 이벤트 | 7일 | 30일 | Hot 7일 → Warm 23일 → Delete |

다운샘플링 전략 (메트릭):
  - 원본: 5분 간격 데이터 (7일 보존)
  - 집계본: 1시간 평균 (90일 보존)
  → 별도 인덱스에 rollup aggregation 결과 저장
  → 저렴한 티어에서 디스크 절약 효과 큼
```

### D1.5-10. 설정 프로파일 (프로젝트별 기본 세팅 템플릿)

```
프로파일 A: 로그 수집 (고처리량, 지연 허용)
{
  "settings": {
    "number_of_shards": 1,
    "number_of_replicas": 0,
    "refresh_interval": "10s",
    "translog.durability": "async",
    "translog.sync_interval": "5s",
    "merge.scheduler.max_thread_count": 1,
    "codec": "default"
  }
}

프로파일 B: 엔티티 데이터 (낮은 처리량, 즉시 가시성)
{
  "settings": {
    "number_of_shards": 1,
    "number_of_replicas": 0,
    "refresh_interval": "1s",
    "translog.durability": "request",
    "codec": "default"
  }
}

프로파일 C: 아카이브 (읽기 전용, 디스크 절약)
{
  "settings": {
    "number_of_shards": 1,
    "number_of_replicas": 0,
    "refresh_interval": "-1",
    "codec": "best_compression"
  }
}
```

- [ ] 저렴한 티어에서 merge.scheduler.max_thread_count를 1로 설정하는 이유를 설명할 수 있다
