## D0단계: 데이터 모델링 의사결정 프레임워크

> **학습 목표**: 주어진 데이터와 쿼리 패턴을 분석하여,
> 최적의 문서 구조(비정규화 깊이, 객체 표현 방식)를 결정하는
> 체계적인 의사결정 프로세스를 익힌다.
>
> **핵심 질문**: "이 데이터를 어떤 형태로 OpenSearch에 넣어야 하는가?"
>
> **전제 조건**: Phase 1의 2단계(매핑 기초) 완료

---

### D0-1. 데이터 모델링 의사결정 프로세스

```
Step 1: 데이터 특성 분석
  - 데이터 관계 (1:1, 1:N, N:M)
  - 변경 빈도 (부모/자식 각각)
  - 데이터 크기 (배열 요소 수, 필드 수)

Step 2: 쿼리 패턴 분석
  - 주요 검색 조건 (어떤 필드로 필터/집계?)
  - 교차 검색 필요 여부 ("배열 내 특정 조합" 검색?)
  - 결과 반환 형태 (부모만? 자식도?)

Step 3: 운영 요구사항 분석
  - 색인 빈도 vs 검색 빈도
  - 업데이트 패턴 (부분 업데이트? 전체 교체?)
  - 데이터 일관성 요구 수준

Step 4: 모델 선택
  → Object / Nested / Flattened / 비정규화 / 별도 인덱스

Step 5: 벤치마크 검증
  → 동일 데이터를 후보 모델로 색인 → 크기/성능 비교
```

### D0-2. 복합 객체 표현 방식 비교: Object vs Nested vs Flattened

**Object (기본)**:
```json
// 매핑
{ "logs": { "type": "object", "properties": { "level": { "type": "keyword" }, "message": { "type": "text" } } } }

// 문서
{ "logs": [
    { "level": "ERROR", "message": "timeout" },
    { "level": "INFO", "message": "retry success" }
] }

// 내부 저장 (평탄화됨!)
{ "logs.level": ["ERROR", "INFO"], "logs.message": ["timeout", "retry success"] }
```

**문제**: `logs.level = ERROR AND logs.message = retry success` 검색하면 매칭됨!
→ 배열 요소 간 경계가 사라짐 (cross-object matching)

**Nested**:
```json
// 매핑
{ "logs": { "type": "nested", "properties": { "level": { "type": "keyword" }, "message": { "type": "text" } } } }

// 내부 저장: 각 배열 요소가 별도 hidden document로 저장
// → logs.level = ERROR AND logs.message = retry success → 매칭 안 됨 ✓
```

**비용**:
- 각 nested 요소가 별도 Lucene document → 문서 수 폭증
- nested 쿼리 필요 (일반 query로 접근 불가)
- 색인/검색 성능 저하

**Flattened**:
```json
// 매핑
{ "metadata": { "type": "flattened" } }

// 문서
{ "metadata": { "any_key_1": "value1", "dynamic_key_99": "value99" } }

// 특성: 모든 값이 keyword처럼 동작, 매핑 폭발 방지
// 제한: range 검색 불가, 풀텍스트 검색 불가, 정확 매칭만
```

**비교 매트릭스**:

| 기준 | Object | Nested | Flattened |
|------|--------|--------|-----------|
| 배열 요소 독립 검색 | ✗ (평탄화) | ✓ | ✗ |
| 매핑 폭발 방지 | ✗ | ✗ | ✓ |
| 색인 성능 | 최고 | 낮음 (문서 수 증가) | 좋음 |
| 검색 성능 | 최고 | 중간 (nested query) | 좋음 |
| 집계 | ✓ | ✓ (nested agg) | △ (keyword만) |
| range/fulltext 검색 | ✓ | ✓ | ✗ |
| 동적 키 대응 | ✗ | ✗ | ✓ |
| 디스크 사용 | 기본 | 높음 (×요소수) | 기본 |

### D0-3. 선택 의사결정 트리

```
배열/객체 필드를 저장해야 한다
    │
    ├── Q1: 배열 요소 간 교차 검색이 문제되는가?
    │     │
    │     ├── NO → Object (기본, 가장 효율적)
    │     │         예: tags: ["a", "b"] → 각 태그를 독립 검색할 필요 없음
    │     │
    │     └── YES → Q2로
    │
    ├── Q2: 배열 요소 수가 많은가? (>100개)
    │     │
    │     ├── YES → 별도 인덱스 분리 고려
    │     │         (nested는 요소당 Lucene doc → 100개면 문서 101개)
    │     │
    │     └── NO → Nested
    │               예: 이슈의 comment 3~5개, 각 comment의 author+date 교차 검색 필요
    │
    ├── Q3: 키가 동적이고 예측 불가한가?
    │     │
    │     ├── YES → Flattened
    │     │         예: 디바이스 메타데이터 (키가 디바이스마다 다름)
    │     │
    │     └── NO → Object 또는 명시적 매핑
    │
    └── Q4: 부모-자식 관계이고, 자식이 독립적으로 검색/업데이트되어야 하는가?
          │
          ├── YES → Join field (parent-child) 또는 별도 인덱스
          │         ⚠️ Join은 성능 비용 매우 높음, 가급적 비정규화 우선
          │
          └── NO → 비정규화 (한 문서에 포함)
```

### D0-4. 비정규화 전략: 어디까지 한 문서에 넣을 것인가

**원칙**: ES는 JOIN이 없다 → 검색/집계 단위 = 문서 단위
→ "함께 검색/집계되는 데이터는 같은 문서에 넣는다"

**비정규화 깊이 결정 기준**:

| 기준 | 비정규화 (한 문서에 포함) | 분리 (별도 인덱스) |
|------|--------------------------|-------------------|
| 함께 검색되는가? | YES → 포함 | NO → 분리 |
| 데이터 크기 | 작다 (< 수KB) | 크다 (> 10KB per item) |
| 변경 빈도 | 드물다 | 잦다 (부모는 안 바뀌는데 자식만 계속 변경) |
| 1:N 비율 | N이 작다 (< 10) | N이 크다 (> 100) |
| 일관성 요구 | 느슨해도 됨 | 강해야 함 (실시간 동기화 필요) |

**예시 — Issue History 시스템**:

```
시나리오: Issue + Comments + Assignee History

방법 A: 완전 비정규화 (한 문서)
{
  "issue_id": "ISS-001",
  "title": "서버 장애",
  "assignee": "홍길동",
  "comments": [
    { "author": "김철수", "text": "확인 중", "at": "..." },
    { "author": "홍길동", "text": "해결됨", "at": "..." }
  ],
  "assignee_history": [
    { "from": "김철수", "to": "홍길동", "at": "..." }
  ]
}
장점: 이슈 검색 시 comment 내용, assignee 이력 모두 한 번에
단점: comment 추가마다 전체 문서 재색인 (_update API)

방법 B: 부분 비정규화 (자주 변하는 것 분리)
- issue-history 인덱스: 이슈 본체 (status, assignee, title)
- issue-events 인덱스: 이벤트 스트림 (comment, 상태변경, 배정변경)

장점: 이벤트 추가 시 append-only, 본체 재색인 불필요
단점: "이 이슈의 comment"를 보려면 2번 검색
```

**의사결정 매트릭스**:
```
비정규화를 선택하라 (if):
  ✓ 검색/집계에서 항상 함께 필요
  ✓ 포함 데이터의 변경이 드물거나 본체와 함께 변경
  ✓ 포함 데이터 크기가 작음
  ✓ 실시간 일관성보다 검색 성능이 우선

분리를 선택하라 (if):
  ✓ 포함 데이터가 독립적으로 검색됨
  ✓ 포함 데이터가 본체보다 훨씬 자주 변경됨
  ✓ 포함 데이터 수가 많거나 계속 증가함
  ✓ 포함 데이터의 실시간 정확성이 중요함
```

### D0-5. Access Pattern → 매핑 결정 프레임워크

```
주어진 쿼리 패턴에서 최적 매핑을 역산하는 프로세스:

Step 1: 쿼리 목록 작성
  - "최근 7일 ERROR 로그에서 service별 건수"
  - "특정 이슈의 모든 이벤트 시간순 조회"
  - "assignee별 평균 해결 시간"

Step 2: 각 쿼리에서 사용되는 필드 역할 분류

  | 필드 | 역할 | 최적 타입 |
  |------|------|---------|
  | 필터 대상 (정확 매칭) | term filter | keyword |
  | 필터 대상 (범위) | range filter | date / numeric |
  | 풀텍스트 검색 | match query | text |
  | 집계 그룹핑 | terms agg | keyword |
  | 집계 수치 | avg/sum agg | numeric |
  | 정렬 기준 | sort | keyword / date / numeric (doc_values 필요) |
  | 결과 표시만 | _source 반환만 | 아무 타입 (index: false 가능) |

Step 3: 필드별 최적 옵션 결정

  | 사용 패턴 | index | doc_values | _source |
  |-----------|-------|-----------|---------|
  | 검색 + 집계 + 표시 | ✓ | ✓ | ✓ |
  | 검색만 (집계/정렬 안 함) | ✓ | ✗ | ✓ |
  | 집계/정렬만 (검색 안 함) | ✗ | ✓ | ✓ |
  | 표시만 (검색/집계 모두 안 함) | ✗ | ✗ | ✓ (또는 enabled: false) |
  | 아예 안 씀 (raw 보관) | ✗ | ✗ | ✓ (enabled: false) |

Step 4: 디스크/메모리 절약 검증
  - 불필요한 index/doc_values 비활성화로 절약되는 양 추정
  - 저렴한 티어에서는 이 최적화가 운영 안정성에 직결
```

### D0-6. 실전 벤치마크 비교 방법론

```
같은 데이터를 서로 다른 모델로 색인하여 비교:

1. 테스트 데이터 준비 (실 데이터 기반, 최소 10만 건)

2. 후보 모델별 인덱스 생성
   - 모델 A: 완전 비정규화 (object)
   - 모델 B: nested 사용
   - 모델 C: 분리 인덱스

3. 측정 항목:
   - 인덱스 크기 (GET /_cat/indices?v)
   - 색인 속도 (bulk API throughput)
   - 대표 쿼리 응답 시간 (5개 이상)
   - 집계 쿼리 응답 시간
   - 업데이트 비용 (부분 수정 시)

4. 비교표 작성:
   | 모델 | 디스크 | 색인 TPS | 검색 p50 | 검색 p99 | 집계 시간 | 업데이트 비용 |
   |------|--------|---------|---------|---------|---------|------------|

5. 트레이드오프 분석:
   - 어떤 모델이 현재 쿼리 패턴에 최적인가?
   - 데이터 증가 시 어떤 모델이 더 잘 스케일되는가?
   - 운영 복잡도는 어떤가?
```

### D0-7. Join Field (Parent-Child) — 왜 거의 쓰지 않는가

```
Join field 특성:
  - 부모-자식 관계를 인덱스 내에서 표현
  - 같은 샤드에 있어야 동작 (routing 필수)
  - has_child / has_parent 쿼리로 관계 검색

왜 피하는가:
  1. 성능 비용 매우 높음 (일반 쿼리 대비 5~10배 느림)
  2. 같은 샤드 강제 → 샤드 불균형 위험
  3. 집계에서 부모-자식 조합이 어려움
  4. 스케일링이 어려움

대안:
  - 대부분의 경우 비정규화가 더 좋음
  - "부모 1 : 자식 수천~수만"이고, 자식 독립 검색이 필수인 경우만 고려
  - 그마저도 별도 인덱스 + application-side join이 보통 더 나음
```

### D0-8. 학습 체크

- [ ] Object vs Nested vs Flattened의 내부 저장 방식 차이를 설명할 수 있다
- [ ] Object의 "평탄화" 문제가 발생하는 구체적 시나리오를 예시로 설명할 수 있다
- [ ] Nested의 비용(문서 수 폭증)을 수치로 계산할 수 있다 (예: 문서 1000개 × nested 5개 = Lucene doc 6000개)
- [ ] 비정규화 vs 분리의 의사결정 기준 5가지를 설명할 수 있다
- [ ] 주어진 쿼리 패턴에서 각 필드의 최적 타입과 옵션(index/doc_values)을 역산할 수 있다
- [ ] Flattened 타입이 적합한 실전 시나리오를 2가지 이상 설명할 수 있다
- [ ] Join field를 거의 쓰지 않는 이유를 성능/운영 관점에서 설명할 수 있다
- [ ] 벤치마크 비교 시 측정해야 할 항목 5가지를 나열할 수 있다
- [ ] 현재 프로젝트의 Issue History 데이터에 대해 비정규화 vs 분리 결정을 내리고 근거를 설명할 수 있다
