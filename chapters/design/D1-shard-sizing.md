## D1단계: 샤드 사이징 & 용량 계획

> **학습 목표**: 데이터 규모, 쿼리 패턴, 노드 리소스를 기반으로
> 적절한 샤드 수와 인덱스 분할 주기를 산출할 수 있다.

### D1-1. 샤드 사이징의 핵심 공식

```
Primary Shard 수 = ceil(예상 인덱스 크기 / 목표 샤드 크기)

목표 샤드 크기:
  - 일반 권장: 10~50GB
  - 시계열 데이터(로그): 20~40GB
  - 검색 중심 데이터: 10~30GB (작을수록 검색 빠름)
  - 저렴한 티어(t3.small): 10~20GB (메모리 제한)
```

### D1-2. 샤드 수 결정에 영향을 주는 요소

```
1. 노드당 메모리 (JVM Heap)
   - 샤드 1개당 약 10~50MB 메모리 오버헤드
   - t3.small (Heap ~1GB) → 최대 50~100개 샤드가 한계
   - 규칙: 노드당 Heap 1GB당 20개 이하 샤드

2. 일간/월간 데이터 증가량
   - 일간 1GB 로그 → 일별 인덱스, shard 1개
   - 일간 100MB 로그 → 주별 또는 월별 인덱스 고려

3. 쿼리 병렬도
   - 검색 시 각 샤드에 스레드 할당
   - 샤드 많으면 병렬도 높지만, 오버헤드도 증가
   - 저렴한 티어(vCPU 2개)에서는 병렬 이점 제한적

4. 보존 기간 × 인덱스 분할 주기
   - 30일 보존, 일별 인덱스 → 30개 인덱스 × shard 수
   - 30일 보존, 주별 인덱스 → 5개 인덱스 × shard 수
```

### D1-3. 용량 계획 프레임워크

```
Step 1: 데이터 수집률 추정
  - 일간 문서 수: N건
  - 평균 문서 크기: X KB (raw JSON)
  - 일간 raw 데이터량: N × X

Step 2: 인덱스 크기 추정 (raw 대비 배율)
  - text 필드 많음: raw × 1.5~2.0 (역색인 + doc_values)
  - keyword/numeric 위주: raw × 1.0~1.3
  - 예: 일간 500MB raw → 인덱스 크기 ~700MB~1GB

Step 3: 인덱스 분할 주기 결정
  - 인덱스 크기 ≤ 목표 샤드 크기 → 일별 인덱스, shard 1개
  - 인덱스 크기 > 목표 샤드 크기 → shard 수 증가 또는 분할 주기 단축

Step 4: 총 샤드 수 검증
  - 총 샤드 수 = (보존 기간 / 분할 주기) × 인덱스당 shard × (1 + replica)
  - 노드당 권장 한도 초과하지 않는지 확인

Step 5: 디스크 용량 검증
  - 총 데이터량 = 인덱스 크기 × (보존 기간 / 분할 주기)
  - 디스크 사용률 85% 이하 유지 (low watermark)
  - 여유 공간: 총 데이터량 / 0.85 ≤ 가용 디스크
```

### D1-4. 오버샤딩의 구체적 비용

```
샤드가 너무 많을 때 (예: 100개 작은 인덱스, 샤드 총 200개):

메모리 비용:
  - 샤드당 Lucene 세그먼트 메타데이터 유지
  - 샤드당 thread pool 할당
  - cluster state에 샤드 정보 유지 (master node 부담)
  
성능 비용:
  - 검색 시 모든 관련 샤드에 fan-out
  - 각 샤드의 결과를 coordinating node에서 merge
  - 작은 샤드(< 1GB)는 오버헤드 비율이 높음

운영 비용:
  - recovery 시간 증가 (샤드 수에 비례)
  - cluster state 업데이트 빈도 증가
  - 모니터링 복잡도 증가
```

### D1-5. 현재 프로젝트 적용 예시

```
시나리오: Issue History (저렴한 티어, t3.small 단일 노드)

- 일간 이슈 생성: ~50건
- 평균 문서 크기: ~2KB
- 일간 데이터량: ~100KB
- 30일 누적: ~3MB
- 1년 누적: ~36MB

→ 인덱스 1개, shard 1개로 충분
→ 월별/연별 분할 불필요 (전체가 1GB 미만)
→ alias로 관리하고 필요 시 reindex

시나리오: 로그 (저렴한 티어, t3.small 단일 노드)

- 일간 로그: ~10만 건
- 평균 문서 크기: ~500B
- 일간 데이터량: ~50MB
- 인덱스 크기(추정): ~65MB (×1.3)
- 30일 보존 시: ~2GB

→ 일별 인덱스, shard 1개
→ 30개 인덱스 × 1 shard = 30 shards (허용 범위)
→ ISM으로 30일 후 삭제
```

### D1-6. 학습 체크

- [ ] 샤드 수 결정 공식(예상 크기 / 목표 샤드 크기)을 적용할 수 있다
- [ ] 노드 메모리 기준 최대 샤드 수를 산출할 수 있다
- [ ] 오버샤딩의 3가지 비용(메모리/성능/운영)을 구체적으로 설명할 수 있다
- [ ] 데이터 수집률 → 인덱스 크기 → 분할 주기 → 총 샤드 수 검증 흐름을 수행할 수 있다
- [ ] 일간 데이터량이 주어졌을 때 인덱스 분할 주기를 결정할 수 있다
