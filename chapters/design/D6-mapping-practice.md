## D6단계: 인덱스 매핑 설계 실습

> **학습 목표**: 요구사항을 분석하여 적절한 인덱스 매핑을 처음부터 설계하고 생성할 수 있다.
>
> **참조**: D1(샤드 사이징), D4(템플릿 구성)

### D6-1. 실습 과제

```
과제: 다음 요구사항을 만족하는 인덱스를 설계하라.

[요구사항]
- 서비스: 디바이스 모니터링 시스템
- 데이터: 디바이스에서 5분마다 전송되는 상태 리포트
- 필드:
  - device_id: 디바이스 고유 ID (정확 매칭 + 집계)
  - device_type: 디바이스 유형 (TV, 냉장고, 세탁기 등)
  - region: 지역 코드 (US, KR, EU 등)
  - timestamp: 리포트 시간
  - metrics: { cpu_usage, memory_usage, disk_usage } (숫자, 범위 검색 + 집계)
  - firmware_version: 펌웨어 버전 (정확 매칭)
  - error_codes: 에러 코드 배열 (정확 매칭 + 집계)
  - raw_payload: 원본 JSON (저장만, 검색/집계 불필요)
  - description: 에러 설명 (풀텍스트 검색)
  
- 쿼리 패턴:
  - "region별, device_type별 평균 cpu_usage"
  - "특정 error_code가 발생한 디바이스 목록"
  - "최근 1시간 디바이스별 최신 리포트"
  - description에서 키워드 검색
  
- 제약:
  - 디바이스 1만 대, 5분 간격 → 일간 ~288만 건
  - 보존 기간: 7일
  - 클러스터: t3.medium 2노드, EBS 100GB 각

[산출물]
1. 매핑 JSON
2. 샤드 수 결정 근거
3. 인덱스 분할 전략 (일별? Data Stream?)
4. 불필요한 기능 비활성화 설정 (index: false 등)
```

### D6-2. 설계 체크포인트

```
✓ 각 필드의 타입 선택 근거를 설명할 수 있는가?
✓ 쿼리 패턴에 맞는 필드 타입을 선택했는가? (keyword vs text)
✓ 불필요한 저장소를 비활성화했는가? (raw_payload → enabled: false)
✓ 데이터량 기반으로 샤드 수를 산출했는가?
✓ Data Stream vs 일반 인덱스 선택 근거가 있는가?
✓ dynamic mapping 전략을 결정했는가? (strict 권장)
```

### D6-3. 다양한 시나리오별 매핑 역설계 워크숍

> 4가지 데이터 유형별로 "쿼리 패턴 → 최적 매핑"을 역산하는 훈련.
> 각 시나리오를 풀고 비교하면, 데이터 유형에 따라 설계 방향이 어떻게 달라지는지 체화된다.

---

**시나리오 A: 로그 (시계열, append-only, 대량)**

```
[서비스]: API Gateway 액세스 로그
[데이터 특성]:
  - 일간 500만 건, 평균 800B
  - 생성 후 수정 없음 (append-only)
  - 최근 3일 집중 검색, 14일 보존

[주요 쿼리]:
  1. "최근 1시간, status_code=500인 로그" (필터)
  2. "서비스별 일간 에러 수 추이" (date_histogram + terms)
  3. "응답시간 p50/p90/p99" (percentiles)
  4. "특정 trace_id로 요청 추적" (정확 매칭)
  5. "에러 메시지에서 키워드 검색" (풀텍스트)

[설계 결정 포인트]:
  - Data Stream vs 일별 인덱스?
  - user_agent (긴 문자열, 거의 검색 안 함) → 어떤 옵션?
  - request_body (매우 큰 필드, 디버깅 시에만 조회) → 어떤 옵션?
  - path (cardinality 높음, 집계에 사용) → 주의점?
  - refresh_interval 전략?
```

---

**시나리오 B: 이벤트 (시계열, 구조화, 중간량)**

```
[서비스]: Alert 발생/해제 이벤트
[데이터 특성]:
  - 일간 10만 건, 평균 1.5KB
  - 알림 발생 → 해제 이벤트 쌍으로 존재
  - 해결 시간(MTTR) 계산 필요

[주요 쿼리]:
  1. "현재 active 알림 목록" (status=fired, resolved_at 없음)
  2. "서비스별 MTTR (발생~해제 시간 차이 평균)" (집계)
  3. "severity=critical이고 30분 이상 미해결" (범위 + 필터)
  4. "특정 rule_id로 발생한 모든 알림 이력" (정확 매칭 + 정렬)
  5. "알림 설명에서 'memory' 키워드 검색" (풀텍스트)

[설계 결정 포인트]:
  - 알림 발생과 해제를 같은 문서로? 별도 문서로?
    → 같은 문서: resolved_at 업데이트 필요 → Data Stream 불가
    → 별도 문서: append-only 가능 → Data Stream 가능, MTTR 계산 복잡
  - alert_context (알림 발생 당시 상세 정보, 2~5KB) → 어떤 옵션?
  - tags (배열, 최대 10개) → nested 필요?
  - Object vs 비정규화 결정 (rule 정보를 이벤트에 포함?)
```

---

**시나리오 C: 메트릭 (시계열, 수치 중심, 초대량)**

```
[서비스]: 디바이스 텔레메트리 (5분 간격 수집)
[데이터 특성]:
  - 디바이스 5만 대 × 5분 간격 = 일간 1440만 건
  - 평균 문서 크기 300B (숫자 필드 위주)
  - 7일 보존 (원본), 90일 보존 (1시간 집계본)

[주요 쿼리]:
  1. "특정 디바이스의 최근 24시간 CPU 추이" (필터 + 정렬)
  2. "region별 평균 메모리 사용률" (terms + avg)
  3. "CPU > 90%인 디바이스 목록" (범위 필터)
  4. "device_type별 시간대별 에러율 히트맵" (date_histogram + terms)
  5. "firmware_version별 평균 배터리 수명 비교" (terms + avg)

[설계 결정 포인트]:
  - 텍스트 필드가 거의 없음 → 역색인 최소화 전략?
  - 필드 대부분이 숫자 → doc_values 중심 설계?
  - device_id (5만 cardinality, 집계에 사용) → 성능 주의점?
  - 다운샘플링 전략: 원본 7일 + 집계본 90일 → 별도 인덱스 설계?
  - 초대량 색인: translog, refresh_interval, bulk 사이즈 전략?
  - _source 전략: 필드가 거의 numeric → _source 필요?
```

---

**시나리오 D: 엔티티 (비시계열, CRUD, 소량)**

```
[서비스]: Alert Rule 설정 관리
[데이터 특성]:
  - 총 ~2000개 규칙, 수시 생성/수정/삭제
  - 변경 이력 추적 필요 (누가, 언제, 뭘 바꿨는지)
  - 규칙 본체는 무기한 보존

[주요 쿼리]:
  1. "active 상태인 특정 서비스의 규칙 목록" (필터 + 정렬)
  2. "조건에 'cpu' 키워드 포함하는 규칙 검색" (풀텍스트)
  3. "최근 수정된 규칙 10개" (정렬)
  4. "담당자별 규칙 수" (terms agg)
  5. "규칙 변경 이력 조회" (특정 rule_id의 히스토리)

[설계 결정 포인트]:
  - Data Stream 불가 (CRUD 필요) → 일반 인덱스 + Alias
  - 변경 이력: 같은 인덱스에 version 필드? 별도 history 인덱스?
  - rule_config (복잡한 JSON 구조, 동적 키 포함) → object? flattened?
  - 매핑 변경 대비: Alias 기반 마이그레이션 설계
  - 소량 데이터: shard 1개, replica 전략?
  - refresh_interval: CRUD 즉시 반영 필요 → 1s 유지?
```

### D6-4. 시나리오별 비교 정리 (풀이 후 작성)

```
| 기준 | 로그 (A) | 이벤트 (B) | 메트릭 (C) | 엔티티 (D) |
|------|---------|----------|----------|----------|
| 인덱스 유형 | Data Stream | ? | Data Stream | 일반 + Alias |
| 분할 전략 | 일별 rollover | ? | 일별 rollover | 단일 인덱스 |
| 주요 필드 타입 | keyword + text | keyword + date | keyword + numeric | keyword + text |
| _source 전략 | 유지 (excludes) | 유지 | 비활성화 검토 | 유지 |
| refresh_interval | 5~10s | 1~5s | 30s | 1s |
| doc_values 전략 | 일부 off | 기본 | 전부 on | 일부 off |
| 핵심 최적화 | 불필요 필드 off | 모델링 결정 | 색인 throughput | Alias 전략 |
```

### D6-5. 학습 체크

- [ ] 요구사항에서 필드별 적절한 타입을 결정할 수 있다
- [ ] 쿼리 패턴을 분석하여 필드 옵션(index, doc_values, enabled)을 결정할 수 있다
- [ ] 데이터량을 기반으로 인덱스 분할 전략과 샤드 수를 산출할 수 있다
- [ ] 완성된 매핑 JSON을 작성하고 각 선택의 근거를 설명할 수 있다
- [ ] 로그/이벤트/메트릭/엔티티 4가지 유형에 대해 각각 다른 설계 방향을 제시할 수 있다
- [ ] 동일 필드라도 데이터 유형에 따라 다른 옵션을 선택하는 이유를 설명할 수 있다
- [ ] 각 시나리오의 핵심 설계 결정 포인트를 식별하고 판단 근거를 설명할 수 있다
