## 3.5단계: 검색 결과 제어 & 페이지네이션

> **학습 목표**: 검색 결과를 그룹핑(collapse)하고, 대량 결과를 효율적으로
> 페이지네이션하며, 반환 데이터를 최적화하는 기법을 익힌다.

### 3.5-1. Collapse — 검색 결과 그룹핑 (중복 제거)

**개념**: 특정 필드 값 기준으로 결과를 그룹핑하여 그룹당 대표 문서 1개만 반환.
SQL의 `GROUP BY`와 유사하지만, 검색 결과 레벨에서 동작.

```json
// "issue_type별로 가장 최근 이슈 1건씩만 보여줘"
GET /issue-history/_search
{
  "query": {
    "bool": {
      "filter": [
        { "range": { "created_at": { "gte": "2025-07-01" } } }
      ]
    }
  },
  "collapse": {
    "field": "issue_type"
  },
  "sort": [{ "created_at": { "order": "desc" } }],
  "size": 10
}
```

**결과**: issue_type이 "bug", "feature", "incident" 등이면 각 타입별 최신 1건만 반환.

**주의사항**:
- `collapse` 필드는 반드시 `keyword` 또는 `numeric` 타입이어야 함
- `text` 필드에는 사용 불가
- `total.value`는 collapse 전 전체 매칭 수 (collapse 후 실제 반환 수와 다름)

### 3.5-2. Collapse + Inner Hits — 그룹 내 상세 문서 조회

```json
// "issue_type별 최신 1건 + 각 그룹의 최근 3건도 함께"
GET /issue-history/_search
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "status": "open" } },
        { "range": { "created_at": { "gte": "2025-07-01" } } }
      ]
    }
  },
  "collapse": {
    "field": "issue_type",
    "inner_hits": {
      "name": "recent_issues",
      "size": 3,
      "sort": [{ "created_at": { "order": "desc" } }],
      "_source": ["issue_id", "title", "created_at", "assignee"]
    }
  },
  "sort": [{ "created_at": { "order": "desc" } }],
  "size": 10
}
```

**결과 구조**:
```json
{
  "hits": {
    "hits": [
      {
        "_source": { "issue_type": "bug", "title": "최신 버그", ... },
        "inner_hits": {
          "recent_issues": {
            "hits": {
              "hits": [
                { "_source": { "issue_id": "ISS-100", "title": "버그1", ... } },
                { "_source": { "issue_id": "ISS-098", "title": "버그2", ... } },
                { "_source": { "issue_id": "ISS-095", "title": "버그3", ... } }
              ]
            }
          }
        }
      },
      {
        "_source": { "issue_type": "incident", "title": "최신 장애", ... },
        "inner_hits": { ... }
      }
    ]
  }
}
```

**다중 inner_hits** — 같은 그룹에서 다른 기준으로 여러 세트 조회:
```json
"collapse": {
  "field": "issue_type",
  "inner_hits": [
    {
      "name": "most_recent",
      "size": 3,
      "sort": [{ "created_at": { "order": "desc" } }]
    },
    {
      "name": "highest_priority",
      "size": 3,
      "sort": [{ "priority_score": { "order": "desc" } }]
    }
  ]
}
```

### 3.5-3. 페이지네이션 전략

**from/size — 기본 페이지네이션 (10,000건 한계)**:
```json
// 1페이지 (1~10)
GET /issue-history/_search
{ "from": 0, "size": 10, "query": { ... } }

// 2페이지 (11~20)
GET /issue-history/_search
{ "from": 10, "size": 10, "query": { ... } }

// ⚠️ from + size > 10,000 이면 에러!
// → 깊은 페이지네이션에 부적합
// → 이유: 각 샤드에서 from+size개를 모두 가져와서 정렬해야 함
```

**search_after — 딥 페이지네이션 (권장)**:
```json
// 첫 페이지
GET /issue-history/_search
{
  "size": 10,
  "query": { "match_all": {} },
  "sort": [
    { "created_at": { "order": "desc" } },
    { "_id": { "order": "asc" } }
  ]
}

// 다음 페이지 — 이전 결과의 마지막 문서의 sort 값 사용
GET /issue-history/_search
{
  "size": 10,
  "query": { "match_all": {} },
  "sort": [
    { "created_at": { "order": "desc" } },
    { "_id": { "order": "asc" } }
  ],
  "search_after": ["2025-07-14T23:59:59Z", "doc-id-xyz"]
}
```

**핵심 규칙**:
- `sort`에 고유한 값(tiebreaker)을 반드시 포함 (`_id` 또는 unique 필드)
- 10,000건 제한 없음
- 단점: "5페이지로 바로 점프" 불가 (순차 탐색만)

**Point in Time (PIT) — 일관된 페이지네이션**:
```json
// 1. PIT 생성 (스냅샷 시점 고정)
POST /issue-history/_search/point_in_time?keep_alive=5m

// 응답: { "id": "pit_id_abc123..." }

// 2. PIT + search_after로 페이지네이션
GET /_search
{
  "pit": { "id": "pit_id_abc123...", "keep_alive": "5m" },
  "size": 10,
  "query": { "match_all": {} },
  "sort": [{ "created_at": { "order": "desc" } }, { "_id": "asc" }],
  "search_after": ["2025-07-14T23:59:59Z", "doc-id-xyz"]
}

// 3. PIT 삭제 (사용 완료 후)
DELETE /_search/point_in_time
{ "id": "pit_id_abc123..." }
```

**PIT의 장점**:
- 페이지네이션 중 인덱스 변경(새 문서 추가/삭제)이 결과에 영향 안 줌
- 일관된 결과 보장
- `search_after`와 함께 사용 권장

**페이지네이션 전략 비교**:

| 방식 | 한계 | 랜덤 접근 | 일관성 | 용도 |
|------|------|---------|--------|------|
| `from/size` | 10,000건 | ✓ | ✗ | UI 페이지네이션 (앞쪽) |
| `search_after` | 없음 | ✗ | △ | 대량 결과 순차 탐색 |
| `PIT + search_after` | 없음 | ✗ | ✓ | 대량 결과 + 일관성 필요 |
| `scroll` (deprecated) | 없음 | ✗ | ✓ | 레거시, PIT로 대체 |

### 3.5-4. 반환 데이터 최적화

**_source filtering — 필요한 필드만 반환**:
```json
GET /issue-history/_search
{
  "query": { "match_all": {} },
  "_source": {
    "includes": ["issue_id", "title", "status", "created_at"],
    "excludes": ["description", "logs"]
  }
}
// 또는 간단히:
{ "_source": ["issue_id", "title", "status"] }
// 또는 완전 비활성화:
{ "_source": false }
```

**docvalue_fields — doc_values에서 직접 읽기 (keyword/numeric/date)**:
```json
GET /issue-history/_search
{
  "_source": false,
  "docvalue_fields": ["issue_id", "status", "created_at"],
  "query": { "term": { "status": "open" } }
}
// → _source 파싱 없이 columnar 저장소에서 직접 읽음
// → keyword, numeric, date 필드에 효과적
```

**fields — 통합 필드 반환 (runtime fields 포함)**:
```json
GET /issue-history/_search
{
  "_source": false,
  "fields": ["issue_id", "title", "status", "resolution_hours"],
  "runtime_mappings": {
    "resolution_hours": {
      "type": "double",
      "script": { "source": "emit(doc['resolution_time_minutes'].value / 60.0)" }
    }
  }
}
```

### 3.5-5. 실전 패턴 조합

**"issue_type별 최신 이슈 + open 상태만 + 페이지네이션"**:
```json
GET /issue-history/_search
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "status": "open" } },
        { "range": { "created_at": { "gte": "2025-07-01", "lte": "2025-07-31" } } }
      ]
    }
  },
  "collapse": {
    "field": "issue_type",
    "inner_hits": {
      "name": "details",
      "size": 5,
      "sort": [{ "created_at": { "order": "desc" } }],
      "_source": ["issue_id", "title", "assignee", "created_at"]
    }
  },
  "sort": [{ "created_at": { "order": "desc" } }],
  "size": 20,
  "_source": ["issue_type", "title", "created_at"]
}
```
