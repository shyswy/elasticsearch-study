## 6.5단계: Lucene 저장 구조 심화 — 설계를 위한 내부 원리

> **학습 목표**: Lucene이 데이터를 디스크에 어떤 형태로 저장하는지 이해하여,
> 인덱스 설계 시 "왜 이 타입이 이 용도에 최적인가", "왜 이 옵션을 끄면 절약되는가"를
> 원리 수준에서 판단할 수 있다.
>
> **핵심 질문**: "내가 문서를 색인하면 디스크에 실제로 무엇이 만들어지는가?"
>
> **전제 조건**: Phase 1의 1~6단계 완료 (특히 2단계의 3가지 저장소 개념)
> **이 단계 이후**: Design Phase(D0~D9)에서 이 지식을 근거로 설계 의사결정

---

### 6.5-1. Segment 내부 파일 구조

```
하나의 Shard = 하나의 Lucene Index
  └── Segment 0
       ├── .tip / .tim   ← Term Dictionary + Term Index (역색인 검색용)
       ├── .doc          ← Posting List (term → doc IDs + freq)
       ├── .pos          ← Positions (term 위치 정보, phrase query용)
       ├── .dvd / .dvm   ← Doc Values (columnar 저장소)
       ├── .fdt / .fdx   ← Stored Fields (_source 원본 JSON)
       ├── .nvd / .nvm   ← Norms (필드 길이 정규화 값, scoring용)
       ├── .kdd / .kdi   ← Points (BKD tree, numeric/geo 범위 검색)
       └── .liv          ← Live Docs (삭제 마킹 비트맵)

핵심 포인트:
  - 각 저장소는 물리적으로 별개 파일
  - 매핑 옵션(index, doc_values, store, norms)은 이 파일들의 생성 여부를 제어
  - 불필요한 파일을 생성하지 않으면 → 디스크 절약 + 메모리 절약
```

### 6.5-2. Inverted Index (역색인) — Posting List 상세

```
역색인의 실제 구조:

Term Dictionary (.tim):
  "error"    → Posting List 위치
  "timeout"  → Posting List 위치
  "server"   → Posting List 위치

Term Index (.tip):
  - Term Dictionary를 빠르게 탐색하기 위한 FST(Finite State Transducer)
  - 메모리에 로드됨 (이게 Heap 소비의 주요 원인 중 하나)

Posting List (.doc):
  "error" → [doc_3, doc_7, doc_15, doc_22, ...]
  
  실제 저장 형태:
  - Delta-encoded: [3, 4, 8, 7, ...] (차이값만 저장 → 압축률 높음)
  - Block 단위 압축 (128개씩 묶어서 PFOR/VInt 인코딩)
  
  포함 정보:
  - Document ID
  - Term Frequency (해당 문서에서 이 term이 몇 번 나오는지)
  - Positions (text 필드, phrase query 지원 시)

검색 흐름:
  match query "error timeout"
    1. Term Index에서 "error" 위치 조회 (FST, O(term길이))
    2. Term Dictionary에서 Posting List 위치 획득
    3. Posting List에서 매칭 doc IDs 획득
    4. "timeout"도 동일하게
    5. 두 Posting List의 교집합/합집합 (skip list 활용)
```

**text vs keyword의 역색인 차이**:
```
text 필드: "Connection timeout error"
  → Analyzer 적용: ["connection", "timeout", "error"]
  → 3개의 term이 역색인에 등록
  → Posting List 3개에 이 문서 ID 추가
  → + Positions 저장 (phrase query 지원)
  → + Norms 저장 (필드 길이 → scoring에 사용)

keyword 필드: "Connection timeout error"
  → Analyzer 미적용: 원본 그대로 1개 term
  → "Connection timeout error" 전체가 하나의 term
  → Posting List 1개에만 등록
  → Positions/Norms 저장 안 함 (정확 매칭만)

결론:
  - text는 역색인 크기가 크다 (term 수 × posting 정보)
  - keyword는 역색인 크기가 작다 (값 1개 = term 1개)
  - 풀텍스트 검색이 불필요한 필드를 text로 매핑하면 디스크 낭비
```

### 6.5-3. Doc Values — Column-Oriented 저장소

```
Doc Values = 문서 ID → 필드 값 매핑 (columnar)

왜 Column-Oriented인가?

Row-oriented (_source):
  doc1: { status: "open", priority: "high", team: "backend" }
  doc2: { status: "resolved", priority: "low", team: "frontend" }
  doc3: { status: "open", priority: "medium", team: "backend" }

Column-oriented (doc_values):
  status:   [doc1→"open", doc2→"resolved", doc3→"open"]
  priority: [doc1→"high", doc2→"low", doc3→"medium"]
  team:     [doc1→"backend", doc2→"frontend", doc3→"backend"]

집계 시 차이:
  "status별 건수" (terms aggregation)
  
  Row-oriented:
    → 모든 문서를 읽어야 함 (전체 JSON 파싱)
    → 불필요한 필드(priority, team)도 읽음
    → I/O 비효율 (전체 데이터 스캔)

  Column-oriented:
    → status 열만 순차 읽기 (다른 필드 안 읽음)
    → 같은 타입 데이터가 연속 → 압축률 높음
    → 순차 I/O → 디스크 캐시 효율 높음
    → CPU SIMD 연산 활용 가능

이게 바로 "doc_values가 집계/정렬에 효율적인 이유"
```

**Doc Values 내부 인코딩 (keyword 필드)**:
```
Ordinals 방식:

원본 값:   ["backend", "frontend", "backend", "backend", "frontend"]
           
Step 1: 고유값 테이블 생성 (Ordinal Table)
  0 → "backend"
  1 → "frontend"

Step 2: 문서별 ordinal 저장
  [doc1→0, doc2→1, doc3→0, doc4→0, doc5→1]

장점:
  - 문자열 반복 저장 안 함 (정수 배열만)
  - 정수 배열은 bit-packing으로 극도로 압축
  - cardinality 낮을수록 (고유값 적을수록) 압축률 높음
  - terms aggregation: ordinal 기준으로 카운트 → 매우 빠름

설계 시 시사점:
  - cardinality 높은 필드 (UUID, 고유 ID 등)는 doc_values 큼
  - cardinality 낮은 필드 (status, level 등)는 doc_values 아주 작음
  - 집계 안 할 고유 ID 필드 → doc_values: false 하면 디스크 절약 효과 큼
```

**Doc Values 내부 인코딩 (numeric 필드)**:
```
integer/long 필드:

저장 방식: GCD compression + bit-packing
  원본: [100, 105, 110, 95, 100]
  GCD = 5, 최솟값 = 95
  정규화: [(100-95)/5, (105-95)/5, ...] = [1, 2, 3, 0, 1]
  → 0~3 범위 → 2bit per value로 충분

float/double:
  - 정밀도에 따라 다양한 인코딩
  - 일반적으로 integer보다 큼

설계 시 시사점:
  - 값 범위가 좁고 규칙적일수록 → doc_values 작음
  - 범위가 넓고 불규칙할수록 → 더 많은 bit 필요
  - 정밀도 불필요하면 scaled_float 고려 (integer로 변환 저장)
```

### 6.5-4. Points (BKD Tree) — Numeric/Date 범위 검색

```
Numeric/Date 필드의 범위 검색은 역색인이 아닌 BKD Tree 사용

BKD Tree (Block KD-Tree):
  - 다차원 공간 분할 구조
  - 1차원(숫자 1개)에서는 사실상 B+Tree와 유사한 범위 검색 구조
  - Leaf 노드에 값들을 정렬된 블록으로 저장

왜 역색인 대신 BKD Tree?
  
  역색인으로 범위 검색:
    range { "age": { "gte": 25, "lte": 35 } }
    → "25", "26", "27", ..., "35" 각각의 Posting List를 합산
    → term 수가 범위에 비례 → 느림 (특히 long/float)

  BKD Tree로 범위 검색:
    → 트리 탐색으로 [25, 35] 범위에 속하는 doc IDs를 한 번에 추출
    → O(log N) + 결과 크기에 비례
    → 범위가 넓어도 효율적

저장 파일: .kdd (data), .kdi (index)

설계 시 시사점:
  - 범위 검색이 주 패턴인 필드 → numeric/date 타입 (BKD)
  - 정확 매칭만 하는 숫자 → keyword도 가능 (하지만 range 못 함)
  - integer vs long: 범위가 int로 충분하면 int (BKD leaf 크기 절반)
  - date: 내부적으로 epoch millis (long)으로 저장 → BKD tree
```

### 6.5-5. Stored Fields (_source) — 원본 JSON 보관

```
Stored Fields (.fdt, .fdx):
  - 원본 JSON을 압축(LZ4/DEFLATE)하여 블록 단위 저장
  - 검색 결과 반환 시 여기서 읽어서 응답

_source의 비용:
  - 디스크: 원본 JSON 크기의 50~70% (압축 후)
  - 읽기: 검색 결과 반환 시 블록 단위 해제 필요
  - 인덱스 전체 크기의 상당 부분을 차지 (보통 30~50%)

_source 최적화 옵션:

1. _source 완전 비활성화:
   { "mappings": { "_source": { "enabled": false } } }
   
   절약: 인덱스 크기 30~50% 감소
   대가:
     - 검색 결과에서 원본 문서 반환 불가 (doc_values/stored로만)
     - update / update_by_query 불가 (원본 없으니 부분 수정 불가)
     - reindex 불가 (원본 없으니 다른 인덱스로 복사 불가)
     - highlight 불가
   
   적합: 집계 전용 인덱스 (문서 본문 조회가 절대 없는 경우)
   위험: 한번 비활성화하면 되돌릴 방법 없음 (reindex 불가이므로)

2. _source includes/excludes (특정 필드만 저장):
   { "mappings": { "_source": { "excludes": ["raw_payload", "debug_info"] } } }
   
   절약: 제외된 필드 크기만큼
   대가: 제외된 필드는 검색 결과에서 반환 불가, reindex 시 복원 불가
   적합: 큰 필드(raw_payload 등)가 조회 불필요할 때

3. Synthetic _source (OpenSearch 2.x+ / ES 8.4+):
   { "mappings": { "_source": { "mode": "synthetic" } } }
   
   원리: _source를 저장하지 않고, doc_values + stored fields에서 재구성
   절약: _source 파일 제거 → 큰 절약
   대가:
     - 필드 순서 보장 안 됨
     - 원본 JSON과 정확히 같지 않을 수 있음 (null 처리 등)
     - 모든 필드에 doc_values 또는 store: true 필요
     - 검색 결과 반환 시 재구성 비용 (약간 느림)
   적합: 디스크 절약이 최우선이고, 원본 형태 보존 불필요한 경우
   주의: OpenSearch 버전 확인 필요 (아직 미지원일 수 있음)

4. Source filtering (쿼리 시점):
   { "_source": { "includes": ["field1", "field2"] } }
   
   이건 저장이 아니라 반환 시 필터링
   → 디스크 절약 아님, 네트워크 트래픽만 절약
   → 설계 결정이 아닌 쿼리 최적화 영역
```

**_source 최적화 의사결정 트리**:
```
이 인덱스의 데이터를 문서 단위로 조회하는가?
  │
  ├── NO (집계만) → _source: false 고려
  │     └── 단, reindex 필요성 없는지 확인 (ISM 삭제 only면 OK)
  │
  └── YES → _source 유지
        │
        ├── 큰 필드가 조회 불필요한가?
        │     ├── YES → excludes: ["big_field"]
        │     └── NO → 기본 유지
        │
        └── 디스크 극단적 절약 필요한가?
              ├── YES + 최신 버전 → synthetic _source 검토
              └── NO → 기본 유지
```

### 6.5-6. Norms — Scoring 보조 정보

```
Norms (.nvd, .nvm):
  - 각 문서의 각 필드에 대한 "필드 길이 인코딩 값"
  - BM25 scoring에서 "짧은 필드에서 매칭 = 더 관련도 높음" 판단에 사용
  - 문서당 1byte (doc 수에 비례)

비활성화:
  { "type": "text", "norms": false }
  또는 keyword는 기본적으로 norms: false

비활성화해도 되는 경우:
  - filter 절에서만 사용 (scoring 불필요)
  - keyword 필드 (기본 꺼져있음)
  - text 필드이지만 scoring 필요 없는 경우 (filter only)

절약량: 문서 수 × 필드 수 × 1byte (대량 데이터에서 유의미)
```

### 6.5-7. 필드 타입별 저장 구조 종합 비교

```
| 필드 타입 | 역색인 | Doc Values | Points(BKD) | Norms | 최적 용도 |
|-----------|--------|-----------|-------------|-------|----------|
| text | Posting List (분석됨) | ✗ | ✗ | ✓ | 풀텍스트 검색 |
| keyword | Posting List (원본) | ✓ (ordinals) | ✗ | ✗ | 정확 매칭 + 집계 |
| integer/long | ✗ (기본) | ✓ (bit-packed) | ✓ (BKD) | ✗ | 범위 검색 + 집계 |
| float/double | ✗ | ✓ | ✓ (BKD) | ✗ | 범위 + 집계 (정밀) |
| date | ✗ | ✓ (epoch ms) | ✓ (BKD) | ✗ | 시간 범위 + 정렬 |
| boolean | Posting List | ✓ | ✗ | ✗ | 필터 + 집계 |
| nested | 별도 Lucene doc | 별도 | ✗ | ✗ | 배열 교차 검색 |
| flattened | Posting List (all values) | ✓ (all values) | ✗ | ✗ | 동적 키 정확 매칭 |

"왜 이 타입이 이 용도에 적합한가" 원리:

keyword가 집계에 좋은 이유:
  → doc_values에 ordinals로 저장 → 카운팅이 정수 배열 순회
  → cardinality 낮으면 메모리 극소량으로 terms agg 가능

numeric이 범위 검색에 좋은 이유:
  → BKD tree로 O(log N) 범위 탐색
  → 역색인으로는 범위의 모든 값을 union해야 함 → O(range_size)

text가 집계에 부적합한 이유:
  → doc_values 기본 비활성 (활성화하면 fielddata → Heap 폭발)
  → 분석된 term으로 집계됨 ("In Progress" → "in", "progress" 각각 카운트)
  → keyword 쓰거나 multi-field의 .keyword 서브필드로 집계
```

### 6.5-8. 매핑 옵션이 물리 파일에 미치는 영향 (종합)

```
PUT /test { "mappings": { "properties": {
  "field_A": { "type": "keyword" },                          
  "field_B": { "type": "keyword", "doc_values": false },      
  "field_C": { "type": "keyword", "index": false },           
  "field_D": { "type": "text", "norms": false },              
  "field_E": { "type": "object", "enabled": false }           
}}}

생성되는 파일:

| 필드 | 역색인(.tim/.doc) | Doc Values(.dvd) | Points(.kdd) | Norms(.nvd) | _source |
|------|------------------|-----------------|-------------|------------|---------|
| field_A | ✓ | ✓ | - | - | ✓ |
| field_B | ✓ | ✗ | - | - | ✓ |
| field_C | ✗ | ✓ | - | - | ✓ |
| field_D | ✓ | - | - | ✗ | ✓ |
| field_E | ✗ | ✗ | - | - | ✓ |

디스크 절약 예시 (100만 문서, 10개 keyword 필드):
  - 모든 필드 기본값: 역색인 10개 + doc_values 10개
  - 집계 안 하는 5개에 doc_values: false → doc_values 파일 5개 생략
  - 검색 안 하는 3개에 index: false → 역색인 파일 3개 생략
  → 전체 인덱스 크기 20~30% 감소 가능 (필드 크기에 따라)
```

### 6.5-9. 실전 설계에의 적용 — "왜 이렇게 설계하는가"

```
Q: "issue_id 필드를 keyword로 하고 doc_values: false로 설정했다. 왜?"
A: issue_id는 정확 매칭 검색용(역색인 필요)이지만, 
   "issue_id별 집계"는 안 한다 (이슈는 고유하니까).
   고유 ID의 doc_values는 cardinality가 높아서 ordinal 압축도 안 되고,
   디스크만 차지한다. → doc_values: false

Q: "response_time_ms를 integer로 했다. float이 더 정밀한데?"
A: 응답 시간은 밀리초 정수로 충분. integer는 BKD tree leaf가 4byte,
   float는 4byte지만 GCD 압축이 안 됨. doc_values도 integer가 
   bit-packing 효율 높음. 정밀도 불필요한 숫자는 항상 integer 우선.

Q: "raw_payload를 enabled: false로 했다. index: false와 뭐가 다르나?"
A: index: false = 역색인 안 만듦 (검색 불가), 하지만 doc_values는 생성됨
   enabled: false = 역색인도, doc_values도, 아무것도 안 만듦. _source에만 존재.
   raw_payload는 검색도, 집계도, 정렬도 안 함 → enabled: false가 최대 절약.
```
