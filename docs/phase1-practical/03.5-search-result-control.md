# 3.5단계: 검색 결과 제어 & 페이지네이션

## 학습 일자
2026-06-09

## 핵심 개념 정리

### Collapse — 검색 결과 그룹핑 (중복 제거)

**정의**: 특정 필드 값 기준으로 검색 결과를 중복 제거하여 그룹당 대표 문서 1개만 반환하는 기능.

**왜 존재하는가**:
- 일반 검색은 정렬 기준에 따라 특정 그룹의 문서만 상위에 몰릴 수 있음
- "각 타입별로 최신 1건씩만 보고 싶다"는 요구사항을 query만으로는 해결 불가
- 클라이언트에서 중복 제거하려면 모든 문서를 가져와야 함 → 비효율

**내부 동작**:
```
1. 일반 검색 수행 (query + sort)
2. 결과에서 collapse 대상 필드의 값을 확인
3. 같은 값을 가진 문서 중 sort 기준 1등만 남김
4. 나머지는 제거
```

**제약사항**:
- collapse 필드는 반드시 **keyword 또는 numeric** 타입이어야 함
- text 필드에 직접 사용 불가 → text는 토큰화되어 원본 값이 보존되지 않으므로 "같은 값인지" 비교 불가능
- text에 collapse 걸려면 multi-field의 keyword sub-field 사용: `"collapse": { "field": "title.keyword" }`
- 응답의 `hits.total.value`는 collapse 전 전체 매칭 수 (실제 반환 문서 수와 다름)
- **collapse와 aggregation은 동시 사용 불가** (ES 7.x 기준)

**사용 예시**:
```json
GET /issue-history/_search
{
  "query": {
    "bool": {
      "filter": [
        { "range": { "created_at": { "gte": "2025-07-01" } } }
      ]
    }
  },
  "collapse": { "field": "issue_type" },
  "sort": [{ "created_at": { "order": "desc" } }],
  "size": 10
}
```

### Collapse + Inner Hits — 그룹 내부까지 조회

**정의**: Collapse로 그룹핑하되, 각 그룹의 내부 문서 N개도 함께 가져오는 기능.

**왜 필요한가**: Collapse만 쓰면 그룹당 1건만 보임. 실제 UI에서는 "해당 타입의 최근 3~5건 목록"도 함께 보여주고 싶은 경우가 많음.

**내부 동작과 비용 구조**:
```
Step 1: Collapse 수행 (그룹당 대표 1건 선택)
Step 2: 각 그룹에 대해 별도의 검색을 추가 수행 (inner_hits)
  → 각 그룹 값을 필터 조건으로 넣은 새로운 검색
  → inner_hits는 독립적인 sort, size, _source를 가질 수 있음
```

**비용**:
- 추가 검색 횟수 = **그룹 수** (그룹마다 1번의 inner search)
- 반환 문서 수 = 그룹 수 × inner_hits size
- 예: 그룹 20개, inner_hits size 5 → 추가 검색 20회, 반환 문서 100건

**왜 Step 1에서 한 번에 못 하나?**:
1. Step 1의 sort 기준과 inner_hits의 sort 기준이 다를 수 있음
2. Step 1에서는 "그룹당 1건"만 남기고 나머지를 이미 버림 — inner_hits에 필요한 그룹 내 다른 문서들은 Step 1 결과에 없음
3. inner_hits의 _source 필터도 별도일 수 있음

**사용 예시**:
```json
GET /issue-history/_search
{
  "query": {
    "bool": {
      "filter": [{ "term": { "status": "open" } }]
    }
  },
  "collapse": {
    "field": "issue_type",
    "inner_hits": {
      "name": "recent_issues",
      "size": 3,
      "sort": [{ "created_at": { "order": "desc" } }],
      "_source": ["issue_id", "title", "created_at", "assignee"]
    }
  },
  "sort": [{ "created_at": { "order": "desc" } }],
  "size": 10
}
```

**다중 inner_hits**: 같은 그룹에서 "최신 3건"과 "우선순위 높은 3건"을 동시에 가져올 수도 있음.

### Collapse vs terms + top_hits 비교

| 기능 | collapse + inner_hits | terms + top_hits |
|------|----------------------|-----------------|
| 위치 | 검색 결과 레벨 | 집계 레벨 |
| 다른 집계와 조합 | ✗ (집계와 동시 사용 제한) | ✓ (다른 metric과 함께 가능) |
| 그룹 수 제어 | size로 전체 결과 수 제한 | terms size로 그룹 수 제한 |
| 추가 통계 | ✗ | ✓ (같은 레벨에 avg, sum 등 추가 가능) |

### 페이지네이션 전략

#### from/size (기본 페이지네이션)

**정의**: SQL의 OFFSET/LIMIT에 해당. from은 건너뛸 문서 수, size는 반환할 문서 수.

**10,000건 한계의 이유** — 분산 시스템에서의 비효율:
```
클러스터: 5개 Shard
요청: from=9000, size=1000 (from+size = 10,000)

실제 동작:
  Shard 0~4: 각각 상위 10,000개 문서를 메모리에 로드 → Coordinating Node로 전송
  Coordinating Node: 5 × 10,000 = 50,000개를 받아서 정렬
  → 9001~10000번째 선택 → 나머지 49,000개 버림

→ from이 커질수록 각 샤드에서 가져오는 데이터가 선형 증가
→ 메모리 폭발 방지를 위해 기본 10,000건으로 제한 (index.max_result_window)
```

**적합한 용도**: UI에서 1~5페이지 정도의 얕은 페이지네이션 + 페이지 번호 클릭(랜덤 점프) 필요 시.

#### search_after (딥 페이지네이션 — 커서 기반)

**정의**: 이전 페이지의 마지막 문서의 sort 값을 기준으로 "그 다음"을 요청하는 방식.

**from/size의 문제를 어떻게 해결하나**:
- 각 샤드는 search_after 값보다 "뒤에 있는" 문서만 size개 반환하면 됨
- from이 아무리 깊어도 각 샤드에서 size개만 처리 → 일정한 비용

**tiebreaker가 필요한 이유**:
- sort 기준(예: created_at)이 같은 문서가 여러 개 있으면, "이 시점 이후"로 커서를 잡을 때 어떤 문서부터 이어갈지 결정할 수 없음
- 문서가 중복되거나 빠질 수 있음
- `_id`를 tiebreaker로 넣으면 고유하게 확정됨

**사용 예시**:
```json
// 첫 페이지
GET /issue-history/_search
{
  "size": 10,
  "query": { "match_all": {} },
  "sort": [
    { "created_at": { "order": "desc" } },
    { "_id": { "order": "asc" } }       // tiebreaker
  ]
}

// 다음 페이지
GET /issue-history/_search
{
  "size": 10,
  "query": { "match_all": {} },
  "sort": [
    { "created_at": { "order": "desc" } },
    { "_id": { "order": "asc" } }
  ],
  "search_after": ["2025-07-14T23:59:59Z", "doc-id-xyz"]
}
```

**제약**: "5페이지로 바로 점프" 불가. 1→2→3→4→5 순차 탐색만.

#### Point in Time (PIT) + search_after

**문제**: 페이지네이션 중 데이터가 변경되면 (새 문서 추가, 기존 문서 삭제/수정):
- 결과가 밀리거나
- 같은 문서가 두 번 나오거나
- 빠지는 문서가 생길 수 있음

이 문제는 from/size, search_after 모두에서 발생 가능 (stateless 페이지네이션 공통 문제).

**PIT의 해결책**: 특정 시점의 인덱스 상태를 "스냅샷"으로 고정해서, 페이지네이션 중 인덱스 변경의 영향을 받지 않게 함.

**PIT의 비용**:
- 세그먼트 참조 유지: PIT 생성 시점의 세그먼트를 "고정" → 해당 세그먼트가 merge되어도 삭제되지 않음
- 검색 자체 성능: PIT 쓴다고 느려지진 않음
- 진짜 비용: keep_alive 동안 "오래된 세그먼트가 안 지워짐" → 디스크 회수 지연
- 해결: keep_alive를 짧게(1~5분) + 끝나면 즉시 삭제

**사용 예시**:
```json
// 1. PIT 생성
POST /issue-history/_search/point_in_time?keep_alive=5m

// 2. PIT + search_after
GET /_search
{
  "pit": { "id": "pit_id_abc...", "keep_alive": "5m" },
  "size": 10,
  "query": { "match_all": {} },
  "sort": [{ "created_at": "desc" }, { "_id": "asc" }],
  "search_after": [...]
}

// 3. PIT 삭제
DELETE /_search/point_in_time
{ "id": "pit_id_abc..." }
```

#### 페이지네이션 전략 비교

| 방식 | 한계 | 랜덤 접근 | 일관성 | 용도 |
|------|------|---------|--------|------|
| from/size | 10,000건 | ✓ | ✗ | UI 페이지네이션 (페이지 번호 클릭) |
| search_after | 없음 | ✗ | △ | 대량 결과 순차 탐색 |
| PIT + search_after | 없음 | ✗ | ✓ | 대량 결과 + 일관성 필요 (Export) |

### 반환 데이터 최적화

#### _source filtering

필요한 필드만 골라서 반환:
```json
{ "_source": ["issue_id", "title", "status", "created_at"] }
// 또는
{ "_source": { "includes": [...], "excludes": [...] } }
// 또는 완전 비활성화
{ "_source": false }
```

#### docvalue_fields

keyword/numeric/date 필드는 Doc Values(열 지향 저장소)에서 직접 읽기:
```json
{
  "_source": false,
  "docvalue_fields": ["issue_id", "status", "created_at"],
  "query": { ... }
}
```

**_source vs docvalue_fields의 차이**:
- _source: 전체 JSON 압축 해제 → 파싱 → 필요 필드 추출
- docvalue_fields: columnar 저장소에서 해당 필드만 직접 읽음 (text 불가)

**docvalue_fields가 이점을 보이는 경우**:
- 필요한 필드가 **전부** doc_values 가능 타입 (keyword/date/numeric/boolean)
- `_source: false`로 _source 압축 해제 자체를 skip할 수 있을 때
- **대량 문서** (수천 건 이상) 처리 시 유의미한 차이

**_source를 한 번이라도 열어야 하면 (doc_values 안 되는 필드가 섞여 있으면)**:
- 어차피 _source 파싱 비용 발생
- _source 파싱 비용은 "몇 개 필드를 꺼내느냐"에 거의 비례하지 않음 (압축 해제는 통째로)
- → 그냥 `_source` filtering으로 통일하는 게 효율적 (섞는 건 오히려 이중 비용)

### doc_values 심화

**자동 생성 규칙**:
- keyword, date, numeric(long/integer/float), boolean, ip → `doc_values: true` (기본, 자동)
- text → `doc_values: false` (기본, 활성화도 불가)
- object (enabled: false) → doc_values 없음

**doc_values가 쓰이는 3가지 상황**:
1. **정렬 (sort)**: `sort: [{ "occurred_at": "desc" }]`
2. **집계 (aggregation)**: terms agg, date_histogram 등
3. **반환 (docvalue_fields)**: _source 대신 직접 읽기

**doc_values를 꺼도 되는 경우**:
- 이 필드를 정렬에 안 쓰고, 집계에도 안 쓰고, docvalue_fields로 조회도 안 함
- 오직 검색(필터)에만 쓰거나 _source로만 반환할 때
- → `"doc_values": false`로 디스크 절약 가능

**주의**: "조회에서 docvalue_fields를 안 쓸 거니까 doc_values 자체를 끄자" → ❌
정렬/집계에 필요할 수 있으므로, 확실히 안 쓸 필드만 꺼야 함.

## 현재 프로젝트 연결

### Issue Trend 테이블 페이지네이션

- UI에 페이지 번호(1, 2, 3...)가 있어서 랜덤 점프 필요 → **from/size** 적합
- workspace별 최근 30~90일 이슈가 10,000건 넘을 가능성 낮음 → from/size 한계 안에서 운영 가능
- sort 기준(occurred_at)이 UPDATE로 변하지 않으므로 PIT 불필요

### Excel/CSV/PDF Export

- 수만 건을 순차 추출 → **PIT + search_after**
- 일관된 스냅샷 보장 필요 (export 중 데이터 변경 영향 방지)

### Summary 집계

- 페이지네이션 불필요: `"size": 0` + aggregation

### search_after로도 일관성 문제가 안 되는 이유 (Issue History 특성)

- Incident Lifecycle: Open(INDEX) → Resolved(UPDATE, 최대 1회) → 끝
- sort 기준(occurred_at)이 UPDATE로 변하지 않음 → 문서 순서 안 흔들림
- 새 문서(INDEX)는 항상 최신 → 이전 페이지에 끼어들지 않음
- UPDATE 빈도가 낮음 (incident당 최대 1번)
- 결론: UI 테이블에서 PIT 없이 from/size만으로 충분

### Incident 인덱스의 doc_values 상태

| 필드 | 타입 | doc_values |
|------|------|-----------|
| issue_instance_id, status, device_id, workspace_id, group_id, group_path, issue_code, issue_type, device_type, os_type, model_name, serial_number, severity, resolved_reason | keyword | ✅ 자동 |
| occurred_at, resolved_at, event_time | date | ✅ 자동 |
| duration_ms | long | ✅ 자동 |
| threshold_value | float | ✅ 자동 |
| is_alertable | boolean | ✅ 자동 |
| description | text (index: false) | ❌ 없음 |
| diagnostics_data | object (enabled: false) | ❌ 없음 |

Issue Trend 테이블에 보여주는 필드(status, occurred_at, resolved_at, group_path, issue_code, device_type, os_type, model_name, serial_number)는 전부 keyword/date → 이론적으로 docvalue_fields만으로 가능.
하지만 10~50건 수준이면 차이 미미 → `_source` filtering으로 통일이 실전적 선택.

## 헷갈렸던 점 & 해결

### 착각: search_after가 Issue Trend 테이블에 적합하다
- **실제**: UI에 페이지 번호 클릭(랜덤 점프)이 있으면 search_after 사용 불가. from/size가 적합.
- **이유**: search_after는 순차 탐색 전용. "5페이지로 바로 점프"가 안 됨.
- **구분**: "일관성 관점에서 PIT 없이도 OK" (맞음) ≠ "페이지네이션 방식으로 search_after를 쓴다" (틀림)

### 착각: inner_hits 비용이 "그룹 수 × size 번 검색"이다
- **실제**: 추가 검색 횟수 = 그룹 수. size는 각 검색에서 반환하는 문서 수.
- **이유**: 한 그룹 내에서는 1번의 검색으로 sort → top N 뽑기를 하니까. size가 3이든 10이든 검색 횟수는 그룹 수.

### 착각: doc_values 안 되는 필드가 섞여 있으면 docvalue_fields + _source를 병행하는 게 효율적이다
- **실제**: 어차피 _source를 한 번이라도 열어야 하면, 거기서 전부 꺼내는 게 더 효율적.
- **이유**: _source 파싱 비용은 "몇 개 필드를 꺼내느냐"에 거의 비례하지 않음. 압축 해제는 통째로 일어남. 이중으로 읽는 게 오히려 손해.

### 착각: 조회에서 docvalue_fields를 안 쓸 거면 doc_values를 꺼도 된다
- **실제**: doc_values는 정렬(sort)과 집계(aggregation)에도 사용됨. 이걸 끄면 해당 필드로 정렬/집계 불가.
- **이유**: doc_values의 3가지 용도: 정렬, 집계, 반환. 반환에 안 쓴다고 꺼버리면 정렬/집계가 깨짐.

## 복습 포인트

1. **Collapse는 검색 결과 레벨의 그룹핑**이다. keyword/numeric 필드에서만 동작하며, text 필드는 토큰화되어 원본 값 비교가 불가능하므로 사용 불가. multi-field의 keyword sub-field로 우회. 응답의 total.value는 collapse 전 전체 매칭 수이므로 실제 반환 수와 다름에 주의. aggregation과 동시 사용 불가.

2. **Inner Hits는 그룹당 1번의 추가 검색**을 수행한다. 그룹 수만큼 추가 검색이 발생하고, 각 검색에서 inner_hits size개를 반환. Step 1(collapse)에서 한 번에 못 하는 이유: collapse 과정에서 그룹당 대표 1건 외 나머지를 이미 버렸고, inner_hits의 sort 기준이 다를 수 있기 때문.

3. **from/size의 10,000건 한계**: 분산 시스템에서 각 샤드가 from+size개를 모두 로드해서 Coordinating Node에 전송하는 구조. 5샤드 × 10,000 = 50,000개를 정렬 후 대부분 버림. from이 커질수록 비효율 선형 증가. 랜덤 페이지 접근이 필요한 UI에 적합.

4. **search_after의 tiebreaker**: 같은 sort 값을 가진 문서가 여러 개일 때 "어디부터 이어갈지" 모호해짐. _id 같은 고유값을 tiebreaker로 넣어서 커서 위치를 확정. 순차 탐색 전용이므로 페이지 점프 불가.

5. **PIT는 세그먼트 스냅샷을 고정**한다. 페이지네이션 중 인덱스 변경의 영향을 받지 않게 함. 비용은 "세그먼트를 안 놓아줘서 디스크 회수 지연"이 핵심. keep_alive를 짧게 잡고 끝나면 즉시 삭제. Issue History처럼 sort 기준 필드가 UPDATE로 변하지 않고 새 문서가 항상 최신인 경우, UI 테이블에서는 PIT 불필요. Export처럼 대량 순차 추출에서만 필요.

6. **반환 데이터 최적화 판단 기준**: (1) 필요 필드 전부 doc_values 가능 + 대량 → `_source: false` + `docvalue_fields`, (2) doc_values 안 되는 필드가 1개라도 섞여 있거나 소량 → `_source` filtering으로 통일. _source 압축 해제는 통째로 일어나므로 필드 수에 거의 비례하지 않음.

7. **doc_values는 자동 생성**되며(keyword/date/numeric/boolean), 정렬·집계·반환 3가지에 쓰임. 반환(docvalue_fields)에 안 쓴다고 꺼버리면 정렬/집계가 깨지므로, "이 필드가 정렬/집계에도 절대 안 쓰일 게 확실할 때"만 꺼야 함.
