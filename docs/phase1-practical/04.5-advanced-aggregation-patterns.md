## 핵심 개념 정리

### top_hits — 그룹별 상위 N개 문서 반환

**정의**: Bucket Aggregation(보통 `terms`) 안에서 사용하는 Metric Aggregation으로, 각 버킷의 실제 문서를 N개 반환한다. `avg`, `sum` 같은 Metric이 "숫자 하나"를 계산하는 반면, `top_hits`는 **문서 자체**를 반환하는 특이한 Metric.

**왜 존재하는가**: `terms` aggregation은 `key`(그룹 이름)와 `doc_count`(문서 수)만 반환함. "issue_code별 건수"는 나오지만, "각 issue_code에서 구체적으로 어떤 이슈가 있는지"는 안 나옴. `top_hits`를 sub-agg로 넣으면 각 버킷에서 실제 문서를 sort/size/source 조건으로 가져올 수 있음.

**핵심 장점 — 다른 Metric과 조합 가능**: 하나의 쿼리로 "그룹별 대표 문서 + 통계 수치"를 모두 뽑을 수 있음. 이것이 `collapse + inner_hits`와의 결정적 차이.

```json
// "issue_code별 top 3 문서 + 평균 해결 시간 + open 건수"
{
  "size": 0,
  "aggs": {
    "by_issue_code": {
      "terms": { "field": "issue_code", "size": 10 },
      "aggs": {
        "latest_issues": {
          "top_hits": {
            "size": 3,
            "sort": [{ "occurred_at": { "order": "desc" } }],
            "_source": ["issue_instance_id", "title", "occurred_at"]
          }
        },
        "avg_duration": {
          "avg": { "field": "duration_ms" }
        },
        "open_count": {
          "filter": { "term": { "status": "open" } }
        }
      }
    }
  }
}
```

**collapse + inner_hits vs terms + top_hits**:

| | collapse + inner_hits | terms + top_hits |
|--|----------------------|-----------------|
| 동작 위치 | 검색 결과 레벨 (hits에서 중복 제거) | 집계 레벨 (aggs 안에서 동작) |
| 다른 집계와 조합 | ❌ 집계와 동시 사용 제한 있음 | ✅ avg, filter 등과 자유롭게 조합 |
| 그룹 수 제어 | size로 전체 문서 수 제한 | terms size로 그룹 수 직접 제한 |
| 추가 통계 | ❌ | ✅ 같은 레벨에 metric agg 추가 가능 |
| 용도 | "대표 문서 목록"만 필요할 때 | "그룹별 통계 + 대표 문서" 동시에 필요할 때 |

### filters aggregation — 다중 조건 동시 집계

**정의**: 직접 정의한 여러 조건을 각각 독립된 버킷으로 만들어 동시에 집계하는 Bucket Aggregation.

**왜 존재하는가**:
- `terms`는 단일 필드의 고유값으로만 그룹핑. 복합 조건("open이면서 critical")은 불가.
- 서로 다른 조건의 건수를 알려면 쿼리를 여러 번 보내야 함.
- `filters`로 원하는 조건을 이름 붙여서 한 번에 모두 집계.

**성능 이점** (단순히 "1회 요청"만이 아님):
1. **인덱스 스캔 1번**: 별도 쿼리 N번이면 N번 스캔하지만, filters는 1번 스캔으로 각 문서가 어떤 필터에 해당하는지 동시에 판단.
2. **Filter Cache 공유**: filter 조건의 결과를 bitset으로 캐시. 반복 실행 시 캐시 히트.
3. **Coordinating Node merge 비용 절감**: 1번만 병합.

```json
GET /issue-history/_search
{
  "size": 0,
  "aggs": {
    "status_breakdown": {
      "filters": {
        "filters": {
          "open": { "term": { "status": "open" } },
          "resolved": { "term": { "status": "resolved" } },
          "critical_open": {
            "bool": {
              "filter": [
                { "term": { "status": "open" } },
                { "term": { "priority": "critical" } }
              ]
            }
          }
        }
      }
    }
  }
}
```

**filters vs terms**:
- terms: 필드의 모든 고유값을 자동으로 그룹핑
- filters: 직접 정의한 복합 조건별로 그룹핑 (이름 지정 가능)

### composite aggregation — 대량 버킷 페이지네이션

**정의**: `terms`의 한계(최대 10,000 버킷)를 넘어, 모든 버킷 조합을 `after` 파라미터로 커서 기반 페이지네이션하며 순회하는 Bucket Aggregation.

**왜 존재하는가**: `terms`는 `size: 10,000`이 한계. 고유값이 50,000개면 전부 가져올 수 없음. 다차원 조합의 모든 경우의 수를 메모리에 올리면 OOM 위험. composite은 한 번에 N개만 로드하고, 다음 페이지를 반복 요청.

**내부 동작**: `search_after`와 유사한 커서 방식. 메모리에 전체 버킷을 올리지 않음.

```json
// 첫 페이지
{
  "size": 0,
  "aggs": {
    "all_combinations": {
      "composite": {
        "size": 100,
        "sources": [
          { "issue_code": { "terms": { "field": "issue_code" } } },
          { "status": { "terms": { "field": "status" } } }
        ]
      },
      "aggs": {
        "avg_duration": { "avg": { "field": "duration_ms" } }
      }
    }
  }
}

// 다음 페이지 — 이전 응답의 after_key 사용
"composite": {
  "size": 100,
  "sources": [ ... ],
  "after": { "issue_code": "NET-03", "status": "resolved" }
}
```

**composite 사용 시점**:
- 전체 버킷을 빠짐없이 순회해야 할 때 (리포트, 배치, 데이터 동기화)
- 다차원 조합을 평탄하게 뽑고 싶을 때 (중첩 깊이 없이)

**composite vs terms 의사결정**:

| 상황 | 추천 |
|------|------|
| 상위 N개만 필요 (대시보드 차트) | terms |
| 고유값 수백 개 이하 + 전부 필요 | terms (size 넉넉히) |
| 고유값 수천~수만 + 전부 필요 | composite |
| 다차원 조합 + 평탄한 결과 | composite |

**성능 비교**:
- terms: 1회 요청으로 빠르지만, size가 크면 메모리 부담
- composite: 메모리 안전하지만 여러 번 요청 필요, 정렬은 자연 순서만 가능

### bucket_sort — 집계 결과 정렬 & 페이지네이션

**정의**: Pipeline Aggregation으로, 다른 sub-agg의 결과값을 기준으로 버킷을 사후 재정렬하거나 페이지네이션한다.

**terms order vs bucket_sort 핵심 차이**:
- `terms order`: 버킷을 **선택(selection)**하는 단계에서 정렬. "어떤 버킷이 반환될지"를 결정.
- `bucket_sort`: **이미 선택된 버킷들**을 사후 재정렬.

**terms order가 참조할 수 있는 것 / 못 하는 것**:
```
✅ 가능: 단순 metric agg — "avg_duration": "desc"
✅ 가능: _count, _key
❌ 불가: filter sub-agg의 _count — "open_count._count": "desc"
❌ 불가: from(offset) 페이지네이션
```

**의사결정**:

| 상황 | 선택 |
|------|------|
| 단순 metric(avg, sum) 기준 정렬 | terms order |
| filter sub-agg의 `_count` 기준 | bucket_sort |
| 정렬 후 페이지네이션(from/size) 필요 | bucket_sort |
| 복수 정렬 기준 조합 | bucket_sort |

```json
// bucket_sort 예시: open 건수 기준 상위 5개
"terms": { "field": "issue_code", "size": 50 },
"aggs": {
  "open_count": { "filter": { "term": { "status": "open" } } },
  "sort_by_open": {
    "bucket_sort": {
      "sort": [{ "open_count._count": { "order": "desc" } }],
      "size": 5
    }
  }
}
```

**주의**: bucket_sort는 "이미 선택된 버킷 내에서" 재정렬하므로, terms의 size를 넉넉히 잡아야 정확한 결과를 얻을 수 있음.

### bucket_selector — 조건부 버킷 필터링

**정의**: Pipeline Aggregation으로, 특정 조건을 만족하지 않는 버킷을 결과에서 제거. SQL의 `HAVING` 절에 해당.

**buckets_path 문법**:
- `buckets_path`는 ES API의 **고정 키명** (Pipeline Agg의 필수 파라미터)
- 값으로 들어가는 건 참조 경로:
  - `"_count"` → 현재 버킷의 doc_count (예약어)
  - `"avg_duration"` → 같은 레벨의 agg 이름
  - `"open_count._count"` → filter sub-agg의 doc_count

```json
"having_slow": {
  "bucket_selector": {
    "buckets_path": {
      "avg_time": "avg_duration"  // "avg_time" = script 변수명 (자유 작명)
                                   // "avg_duration" = 참조할 agg 이름 (존재해야 함)
    },
    "script": "params.avg_time != null && params.avg_time > 7200000"
  }
}
```

### Pipeline Aggregation — cumulative_sum, derivative, percentiles

**Pipeline Agg 개념**: "집계의 결과에 대한 2차 계산". 1차로 생성된 버킷들의 값을 입력으로 받아 추가 계산 수행.

**cumulative_sum — 누적합**:
- 각 버킷까지의 누적 합계
- 예: 6/1: 10건(누적10), 6/2: 15건(누적25), 6/3: 8건(누적33)

**derivative — 변화율 (전 구간 대비 증감)**:
- 이전 버킷 대비 차이값
- 예: 6/1: 10건(null), 6/2: 15건(+5), 6/3: 8건(-7)
- 첫 번째 버킷은 항상 null

**percentiles — 백분위수**:
- 평균만으로는 보이지 않는 분포의 꼬리(tail)를 드러냄
- p50: 1800000(30분), p90: 7200000(2시간), p99: 86400000(24시간)
- → "절반은 30분 안에 해결되지만, 최악의 1%는 24시간 넘게 걸린다"
- SLA 기준은 보통 p95, p99로 잡음 (avg가 아님)

**buckets_path 문법 정리**:
```
"_count"                → 현재 버킷의 doc_count
"sub_agg_name"          → 같은 레벨의 metric agg 결과
"filter_name._count"    → filter sub-agg의 doc_count
"nested_bucket>metric"  → 하위 버킷 안의 metric (> 구분)
```

### filter agg의 역할

**정의**: "이 조건을 만족하는 문서만으로 단일 버킷 1개를 만들어라"

**왜 filter agg를 쓰는가**:
- `term`(단수)은 query 전용이지, agg 타입이 아님
- `terms`(복수) agg는 "모든 고유값별 그룹핑"이지, "특정 조건의 건수"가 아님
- "open인 것만의 건수"를 agg 레벨에서 구하려면 filter agg로 감싸야 함

**query filter vs agg filter 차이**:
- 쿼리의 `bool.filter`: 전체 검색 범위를 제한 (모든 agg에 영향)
- agg의 `filter`: 해당 agg 내에서만 범위 제한 (다른 agg에 영향 없음)

→ 같은 쿼리에서 "전체 건수"와 "open 건수"를 동시에 뽑으려면 agg 레벨의 filter를 써야 함.

**ES에서 이름이 비슷한 3가지**:

| 이름 | 위치 | 역할 |
|------|------|------|
| `term` (단수) | query 안 | "이 필드가 정확히 이 값인 문서를 찾아라" |
| `terms` (복수) | query 안 | "이 값들 중 하나인 문서를 찾아라" |
| `terms` (복수) | aggs 안 | "이 필드의 고유값별로 그룹핑해라" |

### Issue Trend 대시보드에서의 활용

- **top_hits**: "issue_code별 최신 이슈 목록 + 통계"를 단일 쿼리로 추출 → API 응답 1회로 차트+목록 동시 제공

- **percentiles**: "해결 시간의 p50/p90/p99" → SLA 준수 여부 판단

## 핵심 명령어 / API

```
# top_hits (그룹별 상위 문서)
"top_hits": { "size": 3, "sort": [...], "_source": [...] }

# filters (다중 조건 동시 집계)
"filters": { "filters": { "name1": { query }, "name2": { query } } }

# composite (대량 버킷 페이지네이션)
"composite": { "size": 100, "sources": [...], "after": { ... } }

# bucket_sort (사후 정렬)
"bucket_sort": { "sort": [{ "path": { "order": "desc" } }], "size": 5, "from": 0 }

# bucket_selector (조건 필터 = HAVING)
"bucket_selector": { "buckets_path": { "var": "agg_name" }, "script": "params.var > N" }

# cumulative_sum (누적합)
"cumulative_sum": { "buckets_path": "metric_name" }

# derivative (변화율)
"derivative": { "buckets_path": "_count" }

# percentiles (백분위수)
"percentiles": { "field": "duration_ms", "percents": [50, 90, 95, 99] }

# filter agg (단일 조건 버킷)
"filter": { "term": { "status": "open" } }
```

## 헷갈렸던 점 & 해결

### 착각: filter agg 안에 term을 넣는 것은 term aggregation을 쓰는 것
- **실제**: `term`(단수)은 query 전용이지 agg 타입이 아님. agg에 있는 건 `terms`(복수). filter agg 안에 들어가는 `{ "term": {...} }`는 query 조건을 filter agg에 넘긴 것.
- **정리**: `term` = query / `terms` = agg(그룹핑) / `filter` agg = 단일 조건 버킷

### 착각: terms order와 bucket_sort는 같은 일을 한다
- **실제**: terms order = 버킷 **선택** 단계에서 정렬 (어떤 버킷이 반환될지 결정). bucket_sort = 이미 선택된 버킷을 **사후** 재정렬.
- **구체적 차이**: terms order는 filter sub-agg의 `_count` 참조 불가, from(offset) 불가. bucket_sort는 둘 다 가능.

### 체크에서 부족했던 답변: terms order vs bucket_sort 불가능한 케이스
- Q4에서 "불가능한 케이스 2가지"를 구체적으로 말하지 못함.
- **보정**: (1) filter sub-agg의 `_count` 기준 정렬 (2) 페이지네이션(from/size). 이 2가지가 terms order로 불가능해서 bucket_sort가 필요한 핵심 케이스.

## 복습 포인트

1. **top_hits는 "그룹별 문서 반환" Metric Agg**이다. collapse+inner_hits와 달리 다른 metric(avg, filter 등)과 같은 레벨에 자유롭게 조합 가능. "통계 + 대표 문서"를 동시에 원하면 top_hits, "대표 문서만"이면 collapse.

2. **filters는 "직접 정의한 복합 조건별 버킷을 만드는" Bucket Agg**이다. terms(자동 그룹핑)와 달리 복합 조건 가능 + 이름 지정 가능. 성능적으로도 인덱스 스캔 1회 + Filter Cache 공유로 별도 쿼리 N번보다 효율적.

3. **composite은 "전체 버킷을 OOM 없이 페이지네이션으로 순회"하는 Bucket Agg**이다. 주 용도는 전수 리포트/배치. 개수가 적으면 terms가 낫고, composite은 자연 순서 정렬만 가능하다는 제약이 있음. 의사결정: "상위 N개만?" → terms / "전부?" → 10,000 이하면 terms(size 충분히), 초과면 composite.

4. **terms order는 "버킷 선택 기준", bucket_sort는 "사후 재정렬"**이다. terms order는 단순 metric 기준만 가능. filter sub-agg의 _count 기준 정렬, from/size 페이지네이션이 필요하면 bucket_sort. bucket_sort 사용 시 terms size를 넉넉히 잡아야 정확도 보장.

5. **bucket_selector는 SQL의 HAVING**이다. buckets_path로 같은 레벨의 agg 결과를 변수로 받고, script에서 조건 판단. 조건 미충족 버킷은 결과에서 제거. `_count`(예약어)로 doc_count도 참조 가능.

6. **Pipeline Aggregation은 "집계 결과에 대한 2차 계산"**이다. cumulative_sum(누적합), derivative(변화율), bucket_sort(정렬), bucket_selector(필터)가 대표적. buckets_path로 1차 집계 결과를 참조. derivative의 첫 버킷은 항상 null.

7. **filter agg vs query filter**: query의 bool.filter는 전체 범위 제한(모든 agg에 영향), agg의 filter는 해당 agg 내에서만 범위 제한(다른 agg에 영향 없음). "전체 건수"와 "open 건수"를 동시에 뽑으려면 agg 레벨 filter 필수.

8. **ES sub-aggregation vs RDB 서브쿼리**: ES는 1번 스캔으로 모든 레벨의 집계를 동시에 수행(스트리밍 수집기). RDB는 GROUP BY 결과가 요약 행만 반환하므로, 원본 행 상세 정보가 필요하면 재스캔 필요. ES의 비효율은 "중첩 깊이"가 아니라 "버킷 수의 곱"에서 발생(메모리).

9. **percentiles는 평균의 한계를 보완**한다. 평균 2시간이어도 p99가 48시간일 수 있음. SLA 기준은 avg가 아닌 p95/p99로 잡는 것이 표준. 분포의 꼬리(tail)를 드러내는 용도.
